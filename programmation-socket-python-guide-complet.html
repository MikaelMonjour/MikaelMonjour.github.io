Programmation socket en Python : le guide complet

Published: mer. 05 mars 2026
By Mikael Monjour

In tech.

tags: python reseau tech

Vous construisez un outil interne. Un script qui doit envoyer des données à un autre service sur le réseau. Pas un appel HTTP classique — quelque chose de plus bas niveau. Un échange brut entre deux machines, octet par octet.

Vous ouvrez la doc Python, vous tombez sur le module socket, et là : AF_INET, SOCK_STREAM, bind(), listen(), accept()… On dirait du C des années 80.

Bonne nouvelle : c'est exactement du C des années 80 — mais avec Python par-dessus. Et une fois que vous comprenez la logique, tout s'éclaire.

Ce guide couvre tout : de l'echo server le plus simple jusqu'à une application client-serveur complète avec protocole personnalisé, gestion multi-connexion et troubleshooting réseau.

Les sockets, c'est quoi ?

Les sockets sont le mécanisme de base pour envoyer des messages sur un réseau. Ils fournissent une forme de communication inter-processus (IPC) : deux programmes, sur la même machine ou sur deux machines différentes, qui échangent des données.

Le réseau peut être local (votre machine uniquement) ou connecté à l'extérieur — typiquement Internet, via votre fournisseur d'accès.

L'histoire des sockets commence avec ARPANET en 1971. L'API telle qu'on la connaît date de 1983, avec les "Berkeley sockets" du système BSD. Quand le Web a explosé dans les années 90, la programmation réseau a suivi. Serveurs web, navigateurs, applications client-serveur en tout genre — tous utilisent des sockets.

Le plus remarquable : l'API de bas niveau n'a quasiment pas changé depuis. Ce que vous apprenez ici reste valable pour longtemps.

L'API socket de Python en un coup d'œil

Le module socket de Python fournit une interface directe avec l'API socket de Berkeley. Les fonctions Python correspondent aux appels système (syscalls) C :

FonctionRôle
socket()Crée un nouveau socket
bind()Associe le socket à une adresse et un port
listen()Met le socket en mode écoute (serveur)
accept()Accepte une connexion entrante
connect()Se connecte à un serveur distant
connect_ex()Comme connect(), mais retourne un code d'erreur au lieu de lever une exception
send()Envoie des données
recv()Reçoit des données
close()Ferme le socket

Python dispose aussi de modules de plus haut niveau (socketserver, http, smtplib…), mais comprendre les sockets bruts vous donne le contrôle total.

Pourquoi TCP ?

Dans ce guide, nous utilisons socket.SOCK_STREAM, c'est-à-dire le protocole TCP (Transmission Control Protocol). Pourquoi ?

  • Fiable : les paquets perdus sont détectés et retransmis automatiquement.
  • Ordonné : les données arrivent dans l'ordre où elles ont été envoyées.
  • Handshake : un échange initial garantit que les deux côtés sont accessibles.

L'alternative est UDP (socket.SOCK_DGRAM) : plus rapide mais sans garantie de livraison ni d'ordre. Pour la plupart des applications, TCP est le bon choix.

Le flux TCP en image

Voici la séquence d'appels entre un serveur et un client TCP :

Côté serveur :

  1. socket() — crée le socket
  2. bind() — l'associe à une adresse/port
  3. listen() — passe en mode écoute
  4. accept() — attend et accepte une connexion

Côté client :

  1. socket() — crée le socket
  2. connect() — se connecte au serveur (handshake à 3 étapes)

Échange : les deux côtés utilisent send() et recv() pour échanger des données.

Fin : chacun appelle close() pour fermer sa connexion.

Premier exemple : echo server

On commence par le cas le plus simple : un serveur qui renvoie au client exactement ce qu'il reçoit.

Le serveur

#!/usr/bin/env python3

import socket

HOST = '127.0.0.1'  # Adresse de bouclage (localhost)
PORT = 65432        # Port d'écoute (> 1023 pour éviter les privilèges root)

with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:
    s.bind((HOST, PORT))
    s.listen()
    conn, addr = s.accept()
    with conn:
        print('Connecté par', addr)
        while True:
            data = conn.recv(1024)
            if not data:
                break
            conn.sendall(data)

Décortiquons chaque appel :

socket.socket(socket.AF_INET, socket.SOCK_STREAM) crée un objet socket. AF_INET = famille d'adresses IPv4. SOCK_STREAM = type TCP. L'instruction with garantit que le socket sera fermé automatiquement.

bind((HOST, PORT)) associe le socket à l'interface réseau 127.0.0.1 (localhost) et au port 65432. Seuls les processus locaux pourront se connecter. Pour accepter les connexions de n'importe quelle interface, utilisez une chaîne vide '' comme hôte.

listen() active le mode écoute. Le paramètre optionnel backlog définit le nombre maximum de connexions en attente.

accept() bloque jusqu'à ce qu'un client se connecte. Il renvoie un nouvel objet socket conn (pour communiquer avec ce client) et un tuple addr contenant l'adresse IP et le port du client.

Point important : conn est un nouveau socket, distinct du socket d'écoute s. C'est avec conn que vous échangez des données.

La boucle while True lit les données avec conn.recv(1024) (jusqu'à 1024 octets à la fois) et les renvoie avec conn.sendall(data). Quand le client ferme la connexion, recv() renvoie b'' et la boucle s'arrête.

Le client

#!/usr/bin/env python3

import socket

HOST = '127.0.0.1'  # Adresse du serveur
PORT = 65432        # Port du serveur

with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:
    s.connect((HOST, PORT))
    s.sendall(b'Hello, world')
    data = s.recv(1024)

print('Reçu', repr(data))

Le client crée un socket, se connecte au serveur, envoie un message, lit la réponse et l'affiche. Simple.

L'interface de bouclage (loopback)

Quand on utilise l'adresse 127.0.0.1 (ou ::1 en IPv6), les données ne quittent jamais la machine. C'est l'interface de bouclage, aussi appelée "localhost". Elle est utile pour le développement et pour isoler des communications internes.

Dès que vous utilisez une adresse IP réelle (par exemple 10.1.2.3), les données transitent par le réseau externe. Attention à la sécurité.

Gérer plusieurs connexions

Notre echo server a un problème majeur : il ne gère qu'un seul client, puis s'arrête. Dans la vraie vie, un serveur doit accepter des dizaines, des centaines de connexions simultanées.

Plusieurs approches existent : threads, asyncio, ou le vénérable select(). Nous allons utiliser le module selectors de Python, qui fournit une abstraction de haut niveau au-dessus de select() :

"Ce module permet un multiplexage des entrées/sorties de haut niveau et efficace, basé sur les primitives du module select."

L'avantage : pas de threads à gérer, pas de GIL à subir. Et pour une application I/O-bound (qui passe son temps à attendre le réseau), c'est souvent suffisant.

Le serveur multi-connexion

import selectors
import socket
import types

sel = selectors.DefaultSelector()

host, port = '127.0.0.1', 65432
lsock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
lsock.bind((host, port))
lsock.listen()
print('Écoute sur', (host, port))
lsock.setblocking(False)
sel.register(lsock, selectors.EVENT_READ, data=None)


def accept_wrapper(sock):
    conn, addr = sock.accept()
    print('Connexion acceptée de', addr)
    conn.setblocking(False)
    data = types.SimpleNamespace(addr=addr, inb=b'', outb=b'')
    events = selectors.EVENT_READ | selectors.EVENT_WRITE
    sel.register(conn, events, data=data)


def service_connection(key, mask):
    sock = key.fileobj
    data = key.data
    if mask & selectors.EVENT_READ:
        recv_data = sock.recv(1024)
        if recv_data:
            data.outb += recv_data
        else:
            print('Fermeture de la connexion vers', data.addr)
            sel.unregister(sock)
            sock.close()
    if mask & selectors.EVENT_WRITE:
        if data.outb:
            print('Echo', repr(data.outb), 'vers', data.addr)
            sent = sock.send(data.outb)
            data.outb = data.outb[sent:]


while True:
    events = sel.select(timeout=None)
    for key, mask in events:
        if key.data is None:
            accept_wrapper(key.fileobj)
        else:
            service_connection(key, mask)

Les différences clés avec le serveur simple :

  • lsock.setblocking(False) : le socket passe en mode non-bloquant. Plus aucun appel ne suspend l'exécution.
  • sel.register() : on enregistre chaque socket auprès du sélecteur pour surveiller les événements (lecture, écriture).
  • La boucle d'événements : sel.select() bloque jusqu'à ce qu'un socket soit prêt. On distingue ensuite le socket d'écoute (key.data is None) des sockets clients.
  • types.SimpleNamespace : un objet léger pour stocker les données associées à chaque connexion (adresse, tampons d'entrée/sortie).

Le client multi-connexion

import selectors
import socket
import types

sel = selectors.DefaultSelector()
messages = [b'Message 1 du client.', b'Message 2 du client.']


def start_connections(host, port, num_conns):
    server_addr = (host, port)
    for i in range(num_conns):
        connid = i + 1
        print('Connexion', connid, 'vers', server_addr)
        sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        sock.setblocking(False)
        sock.connect_ex(server_addr)
        events = selectors.EVENT_READ | selectors.EVENT_WRITE
        data = types.SimpleNamespace(
            connid=connid,
            msg_total=sum(len(m) for m in messages),
            recv_total=0,
            messages=list(messages),
            outb=b''
        )
        sel.register(sock, events, data=data)


def service_connection(key, mask):
    sock = key.fileobj
    data = key.data
    if mask & selectors.EVENT_READ:
        recv_data = sock.recv(1024)
        if recv_data:
            print('Reçu', repr(recv_data), 'connexion', data.connid)
            data.recv_total += len(recv_data)
        if not recv_data or data.recv_total == data.msg_total:
            print('Fermeture connexion', data.connid)
            sel.unregister(sock)
            sock.close()
    if mask & selectors.EVENT_WRITE:
        if not data.outb and data.messages:
            data.outb = data.messages.pop(0)
        if data.outb:
            print('Envoi', repr(data.outb), 'connexion', data.connid)
            sent = sock.send(data.outb)
            data.outb = data.outb[sent:]


start_connections('127.0.0.1', 65432, 2)

while True:
    events = sel.select(timeout=None)
    for key, mask in events:
        message = key.data
        service_connection(key, mask)

Le client utilise connect_ex() au lieu de connect() pour éviter une exception BlockingIOError immédiate en mode non-bloquant. La connexion se termine en arrière-plan, et select() nous prévient quand le socket est prêt.

Application complète : protocole personnalisé

L'echo server multi-connexion fonctionne, mais il lui manque l'essentiel pour une vraie application : un protocole — c'est-à-dire un format qui définit comment les messages sont structurés, délimités et interprétés.

Le problème des limites de messages

Avec TCP, vous lisez un flux continu d'octets. Il n'y a pas de notion de "message". Quand vous appelez recv(1024), vous pouvez recevoir 1 octet, 500 octets, ou 1024 octets — ce n'est pas garanti.

C'est à vous de définir où commence et où finit un message. Trois approches :

  1. Messages de taille fixe : simple, mais gaspille de la bande passante pour les petits messages.
  2. Délimiteur : un caractère spécial marque la fin (comme \n). Fragile si le délimiteur apparaît dans les données.
  3. En-tête de longueur : on préfixe chaque message avec sa taille. C'est l'approche utilisée par HTTP et celle que nous allons implémenter.

Notre protocole

Chaque message est composé de trois parties :

PartieTailleDescription
En-tête fixe2 octetsLongueur de l'en-tête JSON (entier, big-endian)
En-tête JSONVariableDictionnaire avec les métadonnées : byteorder, content-length, content-type, content-encoding
ContenuVariableLa charge utile (texte JSON ou binaire)

L'approche est hybride : un petit en-tête fixe (2 octets) permet de connaître la taille de l'en-tête JSON variable, qui lui-même contient la taille du contenu.

La classe Message

Toute la logique de lecture, écriture et traitement des messages est encapsulée dans une classe Message. Voici le point d'entrée :

def process_events(self, mask):
    if mask & selectors.EVENT_READ:
        self.read()
    if mask & selectors.EVENT_WRITE:
        self.write()

La méthode read() gère les trois étapes de lecture :

def read(self):
    self._read()

    if self._jsonheader_len is None:
        self.process_protoheader()

    if self._jsonheader_len is not None:
        if self.jsonheader is None:
            self.process_jsonheader()

    if self.jsonheader:
        if self.request is None:
            self.process_request()

Chaque méthode vérifie d'abord qu'elle a assez de données dans le tampon avant de traiter sa partie :

ComposantMéthodeRésultat
En-tête fixe (2 octets)process_protoheader()self._jsonheader_len
En-tête JSONprocess_jsonheader()self.jsonheader
Contenuprocess_request()self.request

Lecture de l'en-tête fixe

def process_protoheader(self):
    hdrlen = 2
    if len(self._recv_buffer) >= hdrlen:
        self._jsonheader_len = struct.unpack(
            '>H', self._recv_buffer[:hdrlen]
        )[0]
        self._recv_buffer = self._recv_buffer[hdrlen:]

struct.unpack('>H', ...) lit un entier non signé de 2 octets en big-endian (ordre réseau). C'est la longueur de l'en-tête JSON qui suit.

Lecture de l'en-tête JSON

def process_jsonheader(self):
    hdrlen = self._jsonheader_len
    if len(self._recv_buffer) >= hdrlen:
        self.jsonheader = self._json_decode(
            self._recv_buffer[:hdrlen], 'utf-8'
        )
        self._recv_buffer = self._recv_buffer[hdrlen:]
        for reqhdr in (
            'byteorder', 'content-length',
            'content-type', 'content-encoding'
        ):
            if reqhdr not in self.jsonheader:
                raise ValueError(f'En-tête requis manquant : "{reqhdr}".')

Traitement de la requête

def process_request(self):
    content_len = self.jsonheader['content-length']
    if not len(self._recv_buffer) >= content_len:
        return
    data = self._recv_buffer[:content_len]
    self._recv_buffer = self._recv_buffer[content_len:]
    if self.jsonheader['content-type'] == 'text/json':
        encoding = self.jsonheader['content-encoding']
        self.request = self._json_decode(data, encoding)
        print('Requête reçue', repr(self.request), 'de', self.addr)
    else:
        self.request = data
        print(f'Requête {self.jsonheader["content-type"]} reçue de',
              self.addr)
    self._set_selector_events_mask('w')

La dernière ligne est cruciale : une fois la requête lue, on passe le sélecteur en mode écriture uniquement. Le serveur n'a plus besoin de lire — il doit maintenant répondre.

Écriture et réponse

def write(self):
    if self.request:
        if not self.response_created:
            self.create_response()

    self._write()


def _write(self):
    if self._send_buffer:
        try:
            sent = self.sock.send(self._send_buffer)
        except BlockingIOError:
            pass
        else:
            self._send_buffer = self._send_buffer[sent:]
            if sent and not self._send_buffer:
                self.close()

La méthode _write() gère le cas où send() ne peut pas tout envoyer d'un coup (tampon réseau plein). Elle attrape l'exception BlockingIOError — le sélecteur rappellera la méthode quand le socket sera de nouveau prêt.

Gestion des erreurs

Dans la boucle d'événements principale, chaque appel à process_events() est entouré d'un try/except :

try:
    message.process_events(mask)
except Exception:
    print(
        'Erreur pour',
        f'{message.addr}:\n{traceback.format_exc()}'
    )
    message.close()

L'appel à message.close() est essentiel : il ferme le socket et le désenregistre du sélecteur. Si un client provoque une erreur, les autres connexions ne sont pas affectées.

Surveillez les statistiques en fin de sortie : perte de paquets et temps de latence.

netstat

Pour voir les sockets actifs et leurs files d'attente :

$ netstat -an | grep 65432
Proto Recv-Q Send-Q Local Address        Foreign Address      (state)

Les colonnes Recv-Q et Send-Q montrent les octets en attente dans les tampons réseau. Des valeurs élevées signifient que l'application n'arrive pas à lire ou écrire assez vite.

Bonnes pratiques

Utilisation des noms d'hôtes

Utilisez des adresses IP numériques plutôt que des noms d'hôtes pour un comportement déterministe. La résolution DNS peut retourner des adresses différentes à chaque appel :

UsageInterfaceRecommandation
ServeurLoopbackUtilisez 127.0.0.1 ou ::1
ServeurEthernetUtilisez une IP spécifique ou '' pour toutes les interfaces
ClientLoopbackUtilisez 127.0.0.1 ou ::1
ClientEthernetIP pour la cohérence, hostname pour le cas typique

SO_REUSEADDR

Ajoutez toujours cette option pour éviter l'erreur "Address already in use" au redémarrage du serveur :

lsock.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)

Sans cette option, après un arrêt du serveur, le port reste en état TIME_WAIT pendant environ 2 minutes.

Fermeture propre des connexions

TCP permet les connexions "semi-ouvertes" : un côté ferme, l'autre reste ouvert. Définissez clairement dans votre protocole comment et quand les connexions se ferment. Pensez aux timeouts pour éviter les connexions zombies.

Appels bloquants vs non-bloquants

Par défaut, les sockets sont en mode bloquant : accept(), connect(), send() et recv() suspendent l'exécution jusqu'à ce qu'ils aient terminé. Pour un serveur multi-connexion, passez en mode non-bloquant avec setblocking(False) et utilisez un sélecteur.

IPv6

Si votre réseau le supporte, testez avec IPv6. La fonction socket.getaddrinfo() simplifie les choses en résolvant automatiquement les adresses IPv4 et IPv6 :

>>> socket.getaddrinfo("example.org", 80, proto=socket.IPPROTO_TCP)
[(<AddressFamily.AF_INET6: 10>, <SocketType.SOCK_STREAM: 1>,
  6, '', ('2606:2800:220:1:248:1893:25c8:1946', 80, 0, 0)),
 (<AddressFamily.AF_INET: 2>, <SocketType.SOCK_STREAM: 1>,
  6, '', ('93.184.216.34', 80))]

Le tuple d'adresse IPv6 contient quatre éléments : (host, port, flowinfo, scopeid), contre deux pour IPv4 : (host, port).
